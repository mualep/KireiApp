# Release-Sliced Implementation Plan

Purpose: replace the current optimistic phase plan with short release slices that depend on a frozen PRD first.

## Scope Note

This is sequencing guidance only. It does not start implementation and it does not rewrite the main PRD.

## Release 0: PRD Freeze Resolution

Focus:
- Resolve all `P0` items from the freeze checklist.
- Confirm and record `UD-01` to `UD-09`.
- Approve the auth/RLS matrix, state-machine truth table, source-of-truth ownership, and audit strategy.

Exit criteria:
- Security boundary accepted.
- State machine accepted.
- Schema ownership accepted.
- Shared unresolved-decision register updated with outcomes or explicit temporary defaults.

## Release 1: Foundation And Landing Baseline

Focus:
- Lock landing architecture around the approved content model.
- Lock bootstrap seeds, owner setup, and default app settings.
- Establish the shared design-token foundation before any landing or admin component implementation.
- Keep public-surface scope limited to current v1 PRD sections.

Execution notes:
- Current repo state is documentation-only. There is no committed app scaffold, `package.json`, lockfile, or `components.json` yet.
- Release 1 therefore starts by creating the Next.js app foundation, then initializing shadcn, then applying the approved tweakcn theme as the base token system.
- Standardize on `pnpm` for the upcoming app scaffold because no package manager is committed yet. The shadcn/tweakcn command for Release 1 should therefore be:
  - `pnpm dlx shadcn@latest add https://tweakcn.com/r/themes/cmmhf5f5t000604l10lbj1ml0`
- Theme install/apply location:
  - Run the command from the future app workspace root, after Next.js scaffold and after shadcn initialization.
  - Preferred app structure for this repo: root `app/` layout, not `src/app/`, to keep the first release simpler.
  - If the scaffold ends up using `src/`, mirror the same plan with `src/app/*` paths instead of `app/*`.

Theme foundation rules:
- Treat the tweakcn theme as the base semantic token system, not as the final visual identity.
- Preserve the frozen PRD direction by keeping Kireiku dark-first, premium, and high-contrast, while retinting semantic tokens as needed after import.
- The fetched tweakcn payload already includes:
  - light and dark token sets
  - `Inter` for sans
  - `JetBrains Mono` for mono
  - semantic sidebar/card/accent/ring tokens
- Keep those structural advantages, but normalize the final token mapping so:
  - landing remains expressive through composition, gradients, imagery, spacing rhythm, and motion
  - admin stays restrained, dense, and task-focused through quieter surfaces and limited accent usage

Expected files changed during Release 1 theme setup:
- App scaffold and base config:
  - `package.json`
  - `pnpm-lock.yaml`
  - `app/layout.tsx` or `src/app/layout.tsx`
  - `app/globals.css` or `src/app/globals.css`
  - `postcss.config.*` if generated by scaffold
  - `tsconfig.json`
- shadcn foundation:
  - `components.json`
  - `lib/utils.ts` or equivalent alias target from shadcn init
- Conditional config files:
  - `tailwind.config.*` only if the scaffold or shadcn setup generates one
  - no separate Tailwind theme file should be invented if Tailwind v4 inline tokens are already used

Dark mode handling:
- Keep class-based theming from the start.
- Default to dark mode for Release 1 foundation to match the frozen PRD direction.
- Support system mode at the provider level if enabled during scaffold, but do not let system mode change the semantic token architecture.
- Use one shared token system across landing and admin; surface-level expression should come from composition, not separate color systems.

Font handling:
- Use `next/font` in the root layout.
- Set `Inter` as the sans variable and `JetBrains Mono` as the mono variable.
- If the imported tweakcn theme writes font tokens into CSS, reconcile them immediately so the final source of truth is:
  - layout-level `next/font` variables
  - global CSS token mapping that points `font-sans` and `font-mono` to those variables or the exact planned stack
- Do not adopt the theme’s serif font as a primary surface font for v1.

Landing vs admin visual handling:
- Landing:
  - keep the imported theme tokens as the baseline
  - allow stronger gradients, larger typography moments, richer section spacing, and more cinematic composition
  - keep brand accents focused around Kireiku’s approved dark-first identity rather than the imported theme’s exact hue balance
- Admin:
  - use the same token base, but stay quieter
  - prefer `bg-background`, `bg-card`, `border-border`, `text-foreground`, `text-muted-foreground`
  - reserve accent/primary emphasis for critical CTA, active nav, status, and focus states
  - avoid marketing-style gradients and decorative effects on task-heavy surfaces

Validation after theme apply:
- Run shadcn project diagnostics:
  - `pnpm dlx shadcn@latest info`
- Verify `components.json` points to the real global CSS path.
- Verify `app/globals.css` or `src/app/globals.css` contains a single source of truth for semantic tokens after import.
- Verify `app/layout.tsx` or `src/app/layout.tsx` mounts `Inter` and `JetBrains Mono` via `next/font`.
- Verify dark mode defaults correctly at the root document level.
- Verify starter landing and starter admin shell both render with the same token system but visibly different register:
  - landing expressive
  - admin restrained
- Verify contrast, focus rings, and basic readability in both light and dark modes before component work continues.

Ordered Release 1 tasks:
1. Create the Next.js app scaffold and standardize on `pnpm`.
2. Initialize shadcn in the app root with the project’s real aliases and global CSS path.
3. Apply the approved tweakcn theme using `pnpm dlx shadcn@latest add https://tweakcn.com/r/themes/cmmhf5f5t000604l10lbj1ml0`.
4. Normalize theme tokens immediately after import so they align with the frozen PRD design direction.
5. Wire `Inter` and `JetBrains Mono` through `next/font` and root layout.
6. Establish class-based dark mode with dark as the default baseline.
7. Validate the theme foundation before building any landing or admin components.
8. Build the landing baseline using the imported token system, not ad-hoc colors.
9. Build the first restrained admin shell using the same token system and semantic tokens.

Exit criteria:
- Foundation work starts only after Release 0 decisions are approved.
- Landing content model is stable enough to seed and test.
- Theme foundation is installed, normalized, and validated before component implementation begins.
- Font, dark mode, and shadcn token plumbing are stable enough that later feature work does not need a parallel theming rewrite.

Focus:
- Cron automation, reconnect/reconciliation, security hardening, and performance hardening.
- Do not include configurable Access Manager UI in shipped v1.
- Keep Future Version items out unless PRD scope is formally amended.

Exit criteria:
- Automation rules are consistent with the frozen truth table.
- Hardening work does not reopen product-scope decisions.

## Unresolved Decision Status

### UD-01 Authenticated `Customer` in v1: keep or remove
- Recommended default: remove authenticated `Customer` from v1.
- Approved PRD decision: Remove authenticated `Customer` from v1. Buyers remain public anonymous users.

### UD-02 Member visibility model: `self-only` vs broader read access
- Recommended default: `Member` is `self-only`.
- Approved PRD decision: `Member` is self-only for tracker, records, and profile.

### UD-03 Worker lifecycle: `hard delete` vs `archive/deactivate-first`
- Recommended default: prefer `archive/deactivate-first` for v1.
- Approved PRD decision: Use archive/deactivate-first for worker lifecycle in v1. Do not ship hard delete UI in v1.

### UD-04 Access Manager UI: keep in v1 or defer
- Recommended default: defer configurable Access Manager UI if freeze scope is still unstable.
- Approved PRD decision: Defer configurable Access Manager UI from v1. Use a documented static permission matrix for v1.

### UD-05 FAQ model: dedicated rows/table vs structured JSON contract
- Recommended default: prefer dedicated FAQ rows/table for validated CRUD.
- Approved PRD decision: Use dedicated FAQ rows/table for validated FAQ CRUD.

### UD-06 Absensi `Custom Text` and future scheduling in v1
- Recommended default: remove `Custom Text` and future scheduling from v1.
- Approved PRD decision: Remove Absensi `Custom Text` and future scheduling from v1.

### UD-07 `ALPHA` correction path: tracker vs absensi/admin correction only
- Recommended default: remove tracker-based `ALPHA -> START`; use absensi/admin correction flow only.
- Approved PRD decision: Remove tracker-based `ALPHA -> START`. `ALPHA` corrections must go through Absensi/Admin correction flow only.

### UD-08 `LEMBUR` accumulation and rounding rule
- Recommended default: accumulate in one internal unit and round only at reporting boundaries.
- Approved PRD decision: Accumulate `LEMBUR` in one internal unit and round only at reporting boundaries.

### UD-09 `BREAK_LATE` idempotency rule
- Recommended default: record `BREAK_LATE` once per break episode after the threshold is crossed.
- Approved PRD decision: Record `BREAK_LATE` once per break episode after the threshold is crossed.
